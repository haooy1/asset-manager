# 开发规范

本文档定义 IT 硬件资产管理系统项目的代码风格、命名约定、Git 工作流和 AI 协作协议，确保代码库长期可维护。

---

## 1. 语言与运行时要求

| 维度 | 规格 |
|------|------|
| **语言** | TypeScript 5.x，全栈统一 |
| **运行时** | Node.js >= 20.9 |
| **包管理器** | pnpm >= 8.0（禁止使用 npm / yarn） |
| **严格模式** | `tsconfig.json` 中 `strict: true`，禁止使用 `any`（特殊情况须用 `// @ts-expect-error reason` 注释理由并加 `// eslint-disable-next-line @typescript-eslint/no-explicit-any`） |

## 3. 命名约定

### 3.1 文件与目录命名

| 类型 | 命名风格 | 示例 |
|------|---------|------|
| **页面文件** | `page.tsx` / `layout.tsx`（Next.js 约定名，不可改） | `page.tsx` |
| **API 路由文件** | `route.ts`（Next.js 约定名，不可改） | `route.ts` |
| **业务逻辑文件** | `camelCase.ts` | `assetService.ts` |
| **类型定义文件** | `camelCase.ts` 或 `types.ts` | `types.ts` |
| **常量文件** | `camelCase.ts` 或 `constants.ts` | `constants.ts` |
| **目录** | `kebab-case` | `asset-detail/` |
| **测试文件** | `*.test.ts` / `*.test.tsx` | `assetService.test.ts` |

### 3.2 代码命名

| 元素 | 命名风格 | 示例 |
|------|---------|------|
| **React 组件** | `PascalCase` | `AssetTable` |
| **函数** | `camelCase` | `getAssetById` |
| **变量** | `camelCase` | `assetCount` |
| **常量** | `SCREAMING_SNAKE_CASE` | `MAX_UPLOAD_SIZE` |
| **类型 / 接口** | `PascalCase` | `Asset`, `ApprovalRequest` |
| **数据库枚举值** | `SCREAMING_SNAKE_CASE` | `IN_USE`, `PENDING_APPROVAL` |
| **布尔变量** | `is` / `has` / `should` 前缀 | `isExpired`, `hasAttachments` |
| **事件处理器** | `handle` + 事件名 | `handleSubmit`, `handleDelete` |
| **Prisma 模型** | `PascalCase` 单数 | `User`, `Asset`, `Approval` |
| **Prisma 字段** | `camelCase` | `createdAt`, `branchId` |

### 3.3 路由命名

| 类型 | 规则 | 示例 |
|------|------|------|
| **页面路由** | 小写 kebab-case | `/login`, `/assets/new` |
| **动态路由** | `[param]` 格式，驼峰 | `/assets/[id]` |
| **API 路由** | 资源名复数 + `[id]` | `/api/assets/[id]` |
| **路由组** | `(groupName)` 括号包裹 | `(dashboard)`, `(auth)` |

## 4. TypeScript 特定规范

### 4.1 类型导入

```typescript
// ✅ 正确：使用 import type 导入仅用于类型的引用
import type { Asset, AssetStatus } from '@/modules/assets/types';
import { getAssetById } from '@/modules/assets/services';

// ❌ 错误：类型和值混在一起导入
import { Asset, getAssetById } from '@/modules/assets/services';
```

### 4.2 interface vs type

| 场景 | 偏好 | 原因 |
|------|------|------|
| 对象形状定义（DTO、Model） | `interface` | 可扩展、可合并声明、IDE 提示更友好 |
| 联合类型、交叉类型 | `type` | `interface` 无法表达联合类型 |
| 函数签名 | `type` | 更简洁，`.d.ts` 中也推荐 |
| 组件 Props | `interface` | 业界惯例，与 React 类型声明一致 |

### 4.3 避免的写法

```typescript
// ❌ 避免：使用 any
const data: any = await fetchAsset(id);

// ✅ 替代方案：明确类型
const data: Asset = await getAssetById(id);
```

```typescript
// ❌ 避免：非空断言（除非有明确注释理由）
const name = asset.user!.name;

// ✅ 替代方案：守卫或可选链
const name = asset.user?.name ?? '未分配';
```

```typescript
// ❌ 避免：类型断言绕过类型检查
const asset = data as Asset;

// ✅ 替代方案：使用 Zod 校验或类型守卫
const asset = assetSchema.parse(data);
```

```typescript
// ❌ 避免：enum（产生额外运行时代码）
enum AssetStatus { IDLE = 'IDLE', IN_USE = 'IN_USE' }

// ✅ 替代方案：const assertion + 字面量联合类型
const ASSET_STATUS = ['IDLE', 'IN_USE', 'MAINTENANCE', 'SCRAPPED'] as const;
type AssetStatus = (typeof ASSET_STATUS)[number];
```

## 5. Next.js / React 特定规范

### 5.1 组件设计原则

- **服务端/客户端边界清晰**：默认使用 Server Component；仅当需要 `useState`、`useEffect`、`onClick` 等交互时，在文件顶部添加 `'use client'`。
- **单一职责**：一个组件只负责一个明确的 UI 区域或交互逻辑，超过 200 行必须拆分。
- **容器/展示分离**：Server Component 作为容器获取数据，Client Component 仅负责展示和交互。
- **Props 类型必须显式声明**：禁止使用 `React.FC<Props>`（已过时），使用 `({ prop }: Props)` 解构模式。

### 5.2 组件文件结构

```typescript
// 1. 声明（如需客户端组件）
'use client';

// 2. 类型导入
import type { Asset } from '@/modules/assets/types';

// 3. 依赖导入（外部库 → 内部模块 → 共享组件 → 相对路径）
import { useRouter } from 'next/navigation';
import { Button } from '@/shared/components/ui/button';
import { getAssetById } from '@/modules/assets/services';

// 4. 类型定义
interface AssetCardProps {
  asset: Asset;
  onDelete?: (id: string) => void;
}

// 5. 组件定义
export function AssetCard({ asset, onDelete }: AssetCardProps) {
  // ...
}
```

### 5.3 导入导出规则

- **命名导出优先**：组件、函数、类型一律使用命名导出（`export function` / `export const` / `export type`），禁止 `export default`，方便 IDE 自动导入和重构。
- **唯一例外**：Next.js 约定的 `page.tsx`、`layout.tsx`、`route.ts` 可以使用默认导出。

### 5.4 导入排序

```typescript
// 按以下顺序分组，组间空一行：
// 1. React / Next.js 核心库
// 2. 第三方库
// 3. 项目内部模块（@/modules/）
// 4. 项目共享代码（@/shared/、@/lib/）
// 5. 相对路径导入
```

### 5.5 Hooks 使用规范

- 自定义 Hook 必须以 `use` 开头命名（如 `useAssets`、`useApprovalList`）
- 自定义 Hook 存放于 `src/shared/hooks/`（通用）或模块内部 `src/modules/[模块名]/hooks.ts`
- API 调用 Hook 使用 React Query (`@tanstack/react-query`) 的 `useQuery` / `useMutation`

## 6. 样式规范

### 6.1 唯一样式方案

- **Tailwind CSS**：所有样式使用 Tailwind 工具类，禁止编写自定义 CSS 文件。
- **shadcn/ui 组件**：通过 `npx shadcn-ui@latest add` 添加，组件源码存于 `src/shared/components/ui/`。
- **动态类名**：使用 `cn()` 工具函数（来自 `@/shared/utils`）合并类名，处理条件样式。
- **响应式**：移动端 H5 适配使用 Tailwind 响应式断点（`sm:` `md:` `lg:` `xl:`），优先移动端设计。

```tsx
// ✅ 正确用法
import { cn } from '@/shared/utils';

<div className={cn('flex items-center gap-2', isActive && 'bg-blue-50')} />

// ✅ 条件样式
<button className={cn(
  'px-4 py-2 rounded-md text-sm',
  variant === 'primary' && 'bg-blue-600 text-white',
  variant === 'outline' && 'border border-gray-300',
  disabled && 'opacity-50 cursor-not-allowed'
)} />
```

### 6.2 禁止的样式写法

```tsx
// ❌ 禁止：内联 style 对象
<div style={{ marginTop: 16, color: 'red' }} />

// ❌ 禁止：CSS Modules 或独立的 .css 文件（Tailwind 是唯一方案）
import styles from './AssetCard.module.css';

// ❌ 禁止：动态拼接类名（Tailwind 无法 tree-shake 识别）
<div className={`text-${color}-500`} />

// ✅ 正确：使用 cn() 或完整类名
<div className={cn(color === 'red' && 'text-red-500', color === 'blue' && 'text-blue-500')} />
```

## 7. Git 工作流

### 7.1 分支策略

独立开发者场景，采用简化 Trunk-Based 开发：

```
main ───────────────────────────────●──●──→
       ↘ feat/xxx ──→ ● ──→ ● ──↗
       ↘ fix/xxx  ──→ ● ──→ ● ──↗
```

| 规则 | 说明 |
|------|------|
| **主分支** | `main`，始终可部署 |
| **功能分支** | 从 `main` 切出，完成后合并回 `main` |
| **分支生命周期** | 1-3 天，避免长期分支导致冲突 |
| **合并方式** | Squash Merge（保持 `main` 提交历史整洁） |

### 7.2 分支命名规范

```
<type>/<short-description>
```

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | `feat/asset-import` |
| `fix/` | Bug 修复 | `fix/approval-status-sync` |
| `refactor/` | 重构 | `refactor/prisma-queries` |
| `docs/` | 文档更新 | `docs/api-spec` |
| `chore/` | 构建/配置 | `chore/dockerfile-optimize` |

### 7.3 提交信息规范 (Conventional Commits)

所有 Git 提交信息**必须**遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<type>(<scope>): <subject>

[body]

[footer]
```

#### 类型 (type)

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变功能） |
| `style` | 代码格式调整（不影响逻辑） |
| `docs` | 文档变更 |
| `test` | 测试相关 |
| `chore` | 构建/工具/依赖 |
| `perf` | 性能优化 |

#### 作用域 (scope)

| scope | 对应范围 |
|-------|---------|
| `assets` | 资产管理模块 |
| `approvals` | 审批流模块 |
| `reminders` | 到期提醒模块 |
| `org` | 组织权限模块 |
| `auth` | 认证相关 |
| `ui` | 共享 UI 组件 |
| `db` | 数据库 schema 或迁移 |
| `config` | 项目配置（ESLint、Prettier、tsconfig 等） |

#### 提交信息规则

- `subject`：中文或英文均可，不超过 72 字符，不加句号。
- `body`：解释**为什么**做这个改动，不是复述代码做了什么。
- `footer`：引用关联的 Issue（如有）。

#### 示例

```
feat(assets): 新增设备批量导入功能

支持通过 CSV 文件批量导入设备信息，包含校验逻辑：
- 自动识别资产编号重复
- 校验必填字段完整性
- 导入失败时返回错误行号和原因
```

```
fix(approvals): 修复审批驳回后状态未同步至资产表

根因：审批服务中驳回逻辑只修改了审批记录状态，未同步
更新资产表的 current_status 字段。

Close #12
```

## 8. 代码注释规范

### 8.1 何时写注释

| 需要注释 | 不需要注释 |
|----------|-----------|
| 非直观的业务规则或计算逻辑 | `// 获取资产列表`（函数名已经说明） |
| 临时解决方案（Workaround）的原因 | `// 设置 loading 为 true`（代码本身足够清晰） |
| 复杂算法的思路说明 | `// 这是一个函数` |
| 跨模块影响的提醒 | 显而易见的参数赋值 |

### 8.2 注释风格

```typescript
// ✅ 解释 "为什么"
// 资产状态必须先置为 SCRAPPED 再删除关联记录，否则外键约束会阻止删除
await db.asset.update({ where: { id }, data: { status: 'SCRAPPED' } });

// ❌ 描述 "做了什么"（代码本身已经清楚表达了）
// 更新资产状态为已报废
await db.asset.update({ where: { id }, data: { status: 'SCRAPPED' } });
```

### 8.3 TODO / FIXME / HACK 标记

```typescript
// TODO(zhangsan): 后续支持按部门筛选后移除此临时过滤逻辑
// FIXME: 并发场景下此查询可能存在竞态条件，需要加悲观锁
// HACK: 麒麟系统下 Prisma binary 需要手动指定 openssl 路径，等官方修复后移除此配置
```

## 9. 错误处理规范

### 9.1 服务端错误处理

统一使用 `NextResponse.json()` 返回结构化错误：

```typescript
// 统一的错误返回类型
type ApiError = {
  error: string;
  message: string;
  details?: Record<string, string[]>; // 字段级校验错误
};

// ✅ 正确：返回结构化错误
import { NextResponse } from 'next/server';

export async function GET(request: Request) {
  try {
    const assets = await getAssetsByBranch(branchId);
    return NextResponse.json({ data: assets });
  } catch (error) {
    if (error instanceof AssetNotFoundError) {
      return NextResponse.json(
        { error: 'NOT_FOUND', message: '资产不存在' },
        { status: 404 },
      );
    }
    console.error('获取资产列表失败:', error);
    return NextResponse.json(
      { error: 'INTERNAL_ERROR', message: '服务器内部错误' },
      { status: 500 },
    );
  }
}
```

### 9.2 客户端错误处理

- API 请求失败时，使用 `toast` 提示用户友好信息（不展示原始错误）。
- 表单提交使用 `try/catch` 包裹，失败时在表单内展示字段级错误。
- 页面级数据加载失败时显示空状态组件，不白屏。

### 9.3 安全原则

- 永远不在 UI 中暴露原始错误堆栈、SQL 错误信息或文件路径。
- 表单校验双重执行：客户端 Zod 校验（即时反馈）+ 服务端 Prisma 约束（安全保障）。
- 生产环境禁用 `console.log`（保留 `console.error` 用于服务端日志）。

## 11. 依赖管理

### 11.1 安装规范

```bash
# ✅ 正确
pnpm add next
pnpm add -D typescript @types/node

# ❌ 禁止：使用 npm 或 yarn
npm install next
yarn add next

# ❌ 禁止：不加 -D 将类型定义安装为生产依赖
pnpm add @types/node
```

### 11.2 依赖审查

- 引入新依赖前优先检查项目中是否已有类似功能的依赖。
- 优先选择社区活跃度高、更新频繁、与 Next.js 生态兼容的库。
- 如果功能简单（如格式化日期），优先自己写工具函数，不引入额外依赖。

### 11.3 版本锁定

- `pnpm-lock.yaml` 必须提交到 Git。
- 手动升级依赖时一次只升级一个包并跑全量测试，不批量升级。

## 13. AI 协作协议

> 本节定义 AI 辅助开发时必须遵守的规则，确保 AI 生成的代码与人工代码保持一致性。

### 13.1 AI 写代码前必读

| 文档 | 目的 |
|------|------|
| `specs/PROJECT-CONTEXT.md` | 建立项目全局认知 |
| `specs/产品概述.md` | 理解业务领域和核心板块 |
| `specs/技术栈.md` | 确认使用的框架和工具 |
| `specs/项目结构.md` | 了解目录结构和模块划分 |
| `specs/开发规范.md`（本文档） | 遵循代码风格和命名约定 |
| 相关模块的现有代码 | 理解当前实现，保持一致性 |

### 13.2 AI 写代码的规则

| 规则 | 说明 |
|------|------|
| **读后写** | 修改文件前必须先 `Read` 该文件 |
| **最小改动** | 只修改任务要求的部分，不做"顺手优化"和"额外重构" |
| **不猜版本** | 安装依赖时使用 `pnpm add` 获取最新版，不手写版本号 |
| **不引入新依赖** | 除非任务明确要求，禁止自动引入第三方库 |
| **不写无意义注释** | 禁止 `// 获取数据`、`// 设置状态` 等复述代码的注释 |
| **不创建多余文件** | 不主动创建 `README.md`、文档等非任务要求的文件 |
| **遵循现有模式** | 新增代码的风格、命名、结构必须与同目录下已有文件一致 |

### 13.3 AI 写完代码后必做

| 步骤 | 操作 |
|------|------|
| **1. 类型检查** | 运行 `pnpm typecheck`，确认无编译错误。如有错误必须全部修复。 |
| **2. Lint 检查** | 运行 `pnpm lint`，确认无 Lint 错误。如有错误必须全部修复。 |
| **3. 依赖方向** | 确认没有违反目录结构中的依赖方向原则（shared 不依赖 modules） |
| **4. 环境变量** | 如有新增环境变量，同步更新 `.env.example` |
| **5. Git 检查** | 确认 `pnpm-lock.yaml` 已更新（如有依赖变更） |

## 14. 目录依赖方向（速查）

```
src/modules/            ← 业务逻辑层（可调用 lib、shared）
       ↓ 调用
src/lib/                ← 基础设施层（可调用 shared）
       ↓ 调用
src/shared/             ← 共享层（无外部依赖）

禁止方向：
  src/shared/    ✗→ src/modules/   （共享层不能依赖业务）
  src/shared/    ✗→ src/lib/       （共享层不能依赖基础设施）
  src/lib/       ✗→ src/modules/   （基础设施不能依赖业务）
```

## 15. 安全编码规范

| 类别 | 规则 |
|------|------|
| **输入校验** | 所有用户输入（表单、URL 参数、请求体）必须通过 Zod Schema 校验后再使用 |
| **注入防护** | 使用 Prisma 参数化查询，禁止拼接 SQL 字符串或 `$queryRawUnsafe` |
| **XSS** | React 默认转义 JSX 中的内容，禁止使用 `dangerouslySetInnerHTML`（如有必要需加 DOMPurify） |
| **密钥管理** | 所有密钥通过环境变量注入，禁止硬编码或提交到 Git |
| **文件上传** | 限制文件大小（≤ 50MB）、校验 MIME 类型白名单、禁止上传可执行文件（`.exe` `.sh` `.bat`） |
| **认证检查** | 所有 API 路由（除登录外）必须通过 NextAuth.js 中间件校验，权限不足返回 403 |
| **CSRF** | NextAuth.js 内置 CSRF token，所有 POST/PUT/DELETE 请求自动校验 |
